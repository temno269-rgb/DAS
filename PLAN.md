1. Что находится в исходном H5

Основной набор данных — матрица:

x[i,c]

где:

i — номер временного отсчёта;
c — номер канала вдоль кабеля;
x — записанное оборудованием значение фазы или её изменения;
при частоте 1000 Гц между отсчётами проходит 0,001 секунды;
соседние каналы расположены примерно через 1,6 метра.

Условно:

                  каналы кабеля
время        c0       c1       c2       c3
0,000 с     x00      x01      x02      x03
0,001 с     x10      x11      x12      x13
0,002 с     x20      x21      x22      x23
...

Важно: пока нет физической калибровки оборудования, это именно значения фазы в единицах прибора. Их нельзя честно называть звуковым давлением, деформацией или децибелами.

2. Чтение исходных значений

Сначала адаптер:

Находит в H5 набор Phase.
Определяет, где расположена временная ось, а где каналы.
Читает частоту, время начала записи и шаг между каналами.
Если числа записаны как упакованные 16-битные значения, расшифровывает их.
Переводит считанный блок в 32-битные числа.

Никакие фильтры не применяются:

пики не вырезаются;
среднее не вычитается;
полосы частот не ограничиваются;
фон из сигнала не удаляется;
пропуски не заменяются выдуманными значениями.

Файл читается блоками примерно по 128–256 МБ. Например:

прочитать блок
→ обработать
→ записать компактный результат
→ освободить память
→ прочитать следующий блок

Это позволяет обработать 5 ГБ, не загружая все 5 ГБ в оперативную память.

3. Две независимые ветви обработки

После чтения работа разделяется на две ветви.

Phase
├── временная ветвь
│   ├── водопад
│   └── активность
│
└── частотная ветвь
    ├── спектрограмма
    ├── спектральная плотность мощности
    └── мощность частотных полос

Я бы выполнял их двумя последовательными проходами по файлу:

Первый проход — временные характеристики.
Второй проход — частотные характеристики.

Это немного увеличивает время первичной обработки, но делает код значительно понятнее и не создаёт конкуренцию за диск и видеопамять. Для нашей задачи это приемлемо.

5. Водопад

Водопад — это временно-пространственная карта:

ось Y → время
ось X → положение вдоль кабеля
цвет → выбранная характеристика

Он не выполняет собственный анализ. Он получает уже рассчитанные статистики окон по 0,25 секунды.

Для каждой ячейки можно получить:

среднеквадратичное значение;
минимум;
максимум;
размах между минимумом и максимумом;
положительный и отрицательный пики;
координаты пиков.

Готовые показатели водопада записываются в temporal.h5:/waterfall/levels/<time_level>/<space_level>/.

Для гарантированного отображения всплесков браузер должен получать пару:

min + max

а не только среднее или один пик. Для линейного временного графика это рисуется как диапазон между минимумом и максимумом.

9. Спектрограмма

Спектрограмма показывает:

ось X → время
ось Y → частота
цвет → мощность

Из общих спектральных кадров для каждой частоты сохраняются:

сумма мощности;
средняя мощность;
максимальная мощность;
число вошедших кадров и каналов;
время и канал максимума.

Общая спектральная мощность, количество наблюдений и координаты максимума записываются в spectral.h5:/spectrum/levels/<time_level>/<space_level>/.

Для обнаружения коротких всплесков основной отображаемой величиной должна быть максимальная мощность. Средняя мощность может сильно ослабить короткое событие.

Проблема группировки восьми каналов

Текущий TODO предлагает объединять по 8 соседних каналов. Тогда:

все восемь входят в сумму;
сильнейший сохраняется как максимум;
сохраняется канал сильнейшего;
но спектры остальных отдельных каналов восстановить уже нельзя.

Здесь необходимо принять явное ограничение.

Оптимальный компромисс:

спектрограмма хранится группами по 8 каналов;
сумма, максимум и координаты максимума сохраняются;
мощность частотных полос хранится отдельно для каждого канала;
интерфейс не обещает полный спектр одного канала после такой группировки.

Если полный спектр любого отдельного канала обязателен, базовая группа должна быть равна одному каналу. Но размер результата тогда может стать сопоставимым с исходным H5 или даже превысить его.

12.1. Нормативная карта хранения результатов

Для всех многоуровневых наборов используется единая схема пути:

/<раздел>/levels/<time_level>/<space_level>/<dataset>

Имена временных уровней:

t_250ms, t_1s, t_5s, t_30s, t_60s, t_300s, t_3600s, t_21600s, t_86400s, t_file

Имена пространственных уровней:

c_1, c_2, c_4, c_8, c_16, c_32, c_64, ... , c_all

Не существующий для конкретного файла уровень не создаётся. Смысл осей, размеры ячеек и список реально созданных уровней записываются в атрибуты соответствующих групп и в manifest.json.

Паспорт и навигация:

manifest.json хранит полный паспорт физического файла, принадлежность серии, временные отношения, состояние обработки, версии схемы и пути к артефактам;
SQLite дублирует только поля, необходимые для быстрого поиска, сортировки и открытия manifest.json;
порядок файлов, разрывы, наложения и идентификатор непрерывного участка записываются в manifest.json каждого затронутого файла и индексируются в SQLite;
данных, существующих только в SQLite и необходимых для восстановления веб-представления, быть не должно.

temporal.h5:/statistics/levels/<time_level>/<space_level>/ хранит достаточные статистики:

12.2. Промежуточные и постоянные значения

Только промежуточными являются:

распакованный блок Phase в 32-битных числах;
остаток неполного окна между блоками;
значения после умножения на окно Ханна;
комплексные коэффициенты частотного преобразования;
рабочий буфер фона активности;
накопители, используемые при построении очередного уровня.

Они существуют только во время ProcessFileJob и после использования освобождаются.

Постоянно записываются:

все достаточные статистики count, sum, sumsq, min и max вместе с координатами экстремумов;
готовые mean, rms, energy и peak_to_peak для водопада;
готовые фон и показатели активности;
power_sum, power_mean, power_max и count для спектральной основы;
готовая спектральная плотность и её характеристики;
готовые мощности частотных полос;
паспорт, временные отношения, состояние и версия результата.

Готовые показатели рассчитываются во время предварительной обработки для каждого созданного уровня. FastAPI не выводит mean, rms, отклонения, спектральные характеристики или мощности полос из достаточных статистик во время пользовательского запроса.

Работа веб-интерфейса
13. Выбор участка кабеля

Внутренне ползунок хранит не метры, а номера каналов:

начальный канал = 100
конечный канал = 400

Метры рассчитываются только для подписи:

d=c\cdot\text{meters_per_channel}

Если шаг равен 1,6 м, ползунок физически не может остановиться между каналами.

14. Выбор времени

Ползунок движется с шагом 1 секунда. Это только ограничение интерфейса.

Внутри выбранного интервала остаются:

временные ячейки по 0,25 секунды;
спектральные кадры с шагом 0,5 секунды;
исходные координаты пиков с точностью одного отсчёта, то есть 0,001 секунды.

Пока пользователь двигает ползунки, запрос не выполняется. Он нажимает «Применить», после чего интерфейс отправляет один запрос с готовыми границами.

15. Что делает сервер после «Применить»

Сервер:

Получает время, каналы и размеры графика.
Читает из manifest.json состояния boundaries и признаки provisional.
Выбирает подходящий готовый уровень.
Читает только пересекающиеся блоки temporal.h5 и spectral.h5.
Возвращает подготовленные числовые массивы вместе с boundary_status, provisional_start и provisional_end, если выбранный диапазон их пересекает.
Браузер обновляет графики.

Сервер при этом не должен:

запускать задачу обработки;
читать исходный Phase;
выполнять частотное преобразование;
вычислять или пересчитывать mean, rms, energy и peak_to_peak;
пересчитывать фон или показатели активности;
пересчитывать спектральную плотность и её характеристики;
пересчитывать мощности полос;
читать исходные участки Phase для сравнения наложений;
вычислять контрольные суммы или определять duplicate и conflict;
объединять ячейки арифметически даже на границах выбранного диапазона.

Допускаются только проверка выбора по manifest.json, выбор готового уровня, чтение срезов, объединение последовательных массивов по абсолютному времени, применение уже сохранённой маски владельца наложения и сериализация ответа. Если граница запроса попадает внутрь готовой ячейки, сервер возвращает эту ячейку вместе с её фактическими границами, не создавая новую характеристику.

16. Обработка файлов и построение серий
16.1. Единица обработки

Одна задача всегда обрабатывает ровно один физический H5-файл.

Если файл меньше 256 МБ — он обычно читается одним блоком.
Если файл больше — читается последовательными блоками не более 256 МБ.
Один вычислительный блок никогда не содержит данные двух физических файлов.
Граница блока внутри файла не является границей математического окна: незавершённые 0,25-секундные окна и хвосты частотных кадров переносятся в следующий блок.

То есть:

200 МБ → один блок обработки
5 ГБ → много блоков до 256 МБ

Но результат в обоих случаях относится к одному физическому файлу.

При записи нельзя просто добавлять данные «в конец». Каждый рассчитанный блок записывается в заранее определённый диапазон:

temporal.h5[time_start:time_end]
spectral.h5[frame_start:frame_end]

В паспорте сохраняются завершённые диапазоны. После сбоя обработка может продолжиться с первого незавершённого диапазона.

16.2. Каталог результатов и обнаружение файлов

Единственная нормативная структура calculated_data и полный порядок работы наблюдателя описаны в разделе «WATCHER PRINCIPE» ниже. Другие варианты структуры каталогов не используются.

Наблюдатель создаёт начальный manifest.json до постановки файла в очередь. ProcessFileJob создаёт temporal.h5 и spectral.h5 в каталоге этого физического файла и обновляет состояние и пути в manifest.json.

Если исходный файл позднее изменился, старая обработка инвалидируется.

17. Серия и непрерывный участок

Серия означает, что файлы описывают одинаковую конфигурацию:

одинаковая частота;
одинаковое количество каналов;
одинаковый шаг между каналами;
одинаковое начало и окончание участка кабеля;
одинаковый порядок осей;
одинаковый тип и единица сигнала.

Однако внутри одной серии могут находиться несколько непрерывных временных участков:

Файлы относятся к одной серии, но между участками находится полуторачасовой разрыв.

Для простой пары файлов вычисляется ожидаемое продолжение:

t_expected = t_start_previous + samples_previous / fs

Затем:

Δt = t_start_next - t_expected

Единая классификация для наблюдателя, фоновых задач, FastAPI и интерфейса:

0 ≤ Δt ≤ 1 секунды — логически непрерывное продолжение;
Δt > 1 секунды — временной разрыв и начало нового непрерывного участка;
Δt < 0 — временное наложение, требующее готового решения OverlapResolverJob.

Логически непрерывное продолжение означает, что следующий файл добавляется в ту же временную последовательность без отдельного предупреждения о разрыве. Исходные абсолютные метки времени при этом не изменяются, пропущенные отсчёты не создаются и значения не интерполируются.

Математически точный стык определяется отдельно: первый отсчёт следующего файла должен по сетке дискретизации следовать сразу за последним отсчётом предыдущего. Допустимая погрешность сравнения временных меток не может превышать один период дискретизации, то есть 0,001 секунды при 1000 Гц. Только такой стык разрешает расчёт окна через границу файлов.

Наблюдатель фиксирует только Δt, отношение файлов и границы наложения. Он не сравнивает фазу и не определяет duplicate или conflict.

17.5. Нормативные правила выбора файлов в веб-интерфейсе

Этот раздел является основным контрактом выбора файлов. Наблюдатель, OverlapResolverJob, сортировка серии, manifest.json, FastAPI и веб-интерфейс должны использовать одинаковые правила раздела 17.

FastAPI формирует список серий и файлов из manifest.json каждого обработанного файла. SQLite используется как быстрый индекс путей и порядка, но значения, показываемые пользователю, берутся из manifest.json.

Для каждого файла интерфейс получает из manifest.json:

технический идентификатор файла и серии;
исходное имя файла;
начало, окончание и продолжительность записи;
начало и окончание участка кабеля;
частоту, число каналов и шаг между каналами;
состояние обработки;
идентификатор непрерывного участка;
отношение к предыдущему файлу и Δt;
границы наложения;
состояние разрешения наложения: pending, duplicate, conflict или failed;
источник конфликтного интервала по умолчанию.

Пользователь самостоятельно отмечает файлы внутри выбранной серии.

Правила проверки выбора:

файлы с 0 ≤ Δt ≤ 1 секунды выбираются совместно как одна последовательность; отдельное уведомление о небольшом временном расхождении не показывается;
при Δt < 0 совместный выбор разрешён только после завершения OverlapResolverJob;
duplicate включается в общую временную шкалу один раз без двойного учёта совпадающих значений;
conflict показывается с предупреждением и указанием выбранного источника; пользователь может переключить источник;
pending или failed блокирует совместное применение перекрывающихся файлов и показывает причину;
при Δt > 1 секунды файлы относятся к разным непрерывным участкам и совместное отображение запрещено;
файлы разных серий нельзя включать в один запрос отображения.

Интерфейс не должен молча изменять выбор пользователя. При недопустимой комбинации он объясняет причину и не позволяет применить выбор, пока пользователь её не исправит.

В начале страницы выводится список серий. Для каждой серии показываются участок кабеля и общий временной диапазон крайних файлов, затем непрерывные участки и входящие в них физические файлы.

18. Жизненный цикл пограничных расчётов

Пограничные расчёты не входят в наблюдатель, обычный ProcessFileJob, FastAPI или браузер. Для них используется отдельная фоновая задача BoundaryFinalizeJob.

BoundaryFinalizeJob применяется только к двум файлам одной серии, которые не накладываются по времени и принадлежат одному логически непрерывному участку. Наложения обрабатывает OverlapResolverJob по разделу 21.

18.1. Состояние до появления соседа

ProcessFileJob полностью рассчитывает каждый физический файл независимо и не ждёт будущих соседей.

Если у файла ещё нет готового предыдущего соседа:

внутренние временные и спектральные результаты файла считаются готовыми;
пограничные частотные кадры, которым нужны отсчёты соседнего файла, не создаются;
если предыдущий файл уже известен, но ещё не готов, показатели активности в начале текущего файла помечаются provisional;
если предыдущий файл не известен и файл пока является первым в участке, фон активности считается сброшенным, activity_status получает ready_reset и может быть инвалидирован при позднем появлении более раннего соседа;
для неизвестного будущего следующего файла правая граница получает status: awaiting_neighbor.

Файл можно просматривать до финализации границы. FastAPI возвращает признак provisional только для диапазона, который действительно ожидает известного предыдущего соседа.

18.2. Создание BoundaryFinalizeJob

После изменения порядка файлов наблюдатель записывает отношение соседей в их manifest.json. Сам наблюдатель пограничные вычисления не запускает.

Диспетчер фоновых задач создаёт BoundaryFinalizeJob, когда одновременно выполнены условия:

оба файла имеют processing_status: ready;
файлы принадлежат одной серии;
следующий файл действительно следует за предыдущим в каноническом порядке серии;
0 ≤ Δt ≤ 1 секунды;
для этой пары и текущих версий исходных файлов ещё нет готового результата.

Последовательность:

новый сосед обнаружен
→ оба ProcessFileJob готовы
→ BoundaryFinalizeJob
→ проверка временной сетки
→ финализация активности
→ при точном стыке расчёт пограничных частотных кадров
→ обновление затронутых уровней
→ обновление версий
→ инвалидирование кэша

Если Δt > 1 секунды, BoundaryFinalizeJob не создаётся: фон активности второго файла окончательно считается сброшенным, а граница получает status: skipped_gap.

Если Δt < 0, BoundaryFinalizeJob не создаётся: применяется OverlapResolverJob.

18.3. Проверка математической непрерывности

Логическая непрерывность до одной секунды и математически точный стык являются разными условиями.

Точный стык существует, если первый отсчёт следующего файла по сетке дискретизации непосредственно продолжает предыдущий файл. Погрешность представления временной метки допускается не больше одного периода дискретизации, но пропущенного отсчёта внутри объединяемого окна быть не должно.

BoundaryFinalizeJob записывает:

delta_t_seconds;
is_exact_sample_continuity;
expected_next_sample_time;
actual_next_sample_time.

Если 0 < Δt ≤ 1 секунды и реальный отсчёт отсутствует, файлы остаются одним логическим участком, но частотный кадр через границу не строится и отсутствующее значение не создаётся.

18.4. Пограничные частотные кадры

При точном стыке задача находит только те секундные окна с шагом 0,5 секунды, которые пересекают границу файлов и ещё не были рассчитаны обычными ProcessFileJob.

Для каждого такого окна задача:

читает минимально необходимый хвост Phase предыдущего файла и начало Phase следующего файла;
применяет то же окно Ханна, масштабирование и частотное преобразование, что и основная спектральная ветвь;
получает spectrum, psd и band_power;
записывает базовый результат в spectral.h5 файла-владельца;
перестраивает только родительские временные и пространственные уровни, в которые попал новый кадр.

Правило владения:

центр кадра вычисляется как frame_center = frame_start + frame_length / 2;
кадр принадлежит файлу, полуоткрытый диапазон [start, end) которого содержит frame_center;
если центр совпадает с start следующего файла, владельцем является следующий файл;
один frame_id может существовать только в одном spectral.h5.

В manifest.json владельца сохраняются frame_id, frame_start, frame_end, frame_center и идентификаторы обоих исходных файлов.

Если точного стыка нет, spectral_status получает значение skipped_non_continuous и пограничные частотные кадры не создаются.

18.5. Восстановление фона активности

Для начала следующего файла BoundaryFinalizeJob не читает исходный Phase повторно. Он использует достаточные статистики из temporal.h5:/statistics/levels/t_250ms/c_1/:

sum;
sumsq;
count;
min;
max.

Задача идёт назад по готовым файлам того же непрерывного участка и набирает до 60 секунд реальной истории. Она останавливается после набора 60 секунд либо в начале непрерывного участка. Ограничиваться только непосредственным соседом нельзя, потому что один физический файл может быть короче 60 секунд.

Затем задача использует начальные окна следующего файла и заново рассчитывает background_mean, background_std, deviation_positive, deviation_negative и normalized_deviation для его начальной части, пока каждому окну не станет доступна требуемая предыдущая история.

При 0 < Δt ≤ 1 секунды фон продолжается через логический стык, но отсутствующий интервал не заполняется: он добавляет ноль к count и ничего к sum и sumsq.

Обновляются только:

temporal.h5:/activity/levels/t_250ms/c_1/ для затронутых окон;
родительские временные и пространственные уровни activity, зависящие от этих окон;
координаты изменившихся экстремумов активности.

Разделы statistics и waterfall не изменяются. После успешного обновления activity_status становится ready, а provisional снимается только с фактически пересчитанного диапазона.

18.6. Что означает обновление уровней серии

Отдельные series_temporal.h5 и series_spectral.h5 по принятой архитектуре не создаются.

Поэтому «обновить уровни серии» означает:

обновить новые пограничные кадры и их родительские уровни в spectral.h5 файла-владельца;
обновить затронутые уровни activity в temporal.h5 следующего файла;
увеличить artifact_revision в manifest.json изменённых файлов;
увеличить series_revision в SQLite;
сделать недействительными кэшированные ответы серии, построенные на предыдущей series_revision.

series_revision является технической версией кэша и индекса. Она не заменяет сведения manifest.json и не является источником пользовательских данных.

Ключ кэша должен включать series_id и series_revision. После увеличения версии старые записи больше не выдаются, а их физическое удаление может выполняться позднее в фоне.

18.7. Надёжность и повторный запуск

BoundaryFinalizeJob имеет уникальный ключ:

series_id + previous_file_id + next_file_id + обе source_revision + background_history_signature + processing_schema_version

Состояния пограничной записи и задачи:

awaiting_neighbor — предыдущий сосед ещё неизвестен;
awaiting_files — отношение известно, но один из файлов ещё не готов;
pending — оба файла готовы и задача поставлена в очередь;
running — задача выполняется;
ready — все требуемые пограничные результаты записаны;
skipped_gap — разрыв больше секунды, расчёт не требуется;
failed — задача завершилась ошибкой.

Перед изменением temporal.h5 или spectral.h5 задача получает блокировку соответствующего артефакта. Запись результатов и всех зависимых уровней завершается до установки status: ready в manifest.json.

Задача должна быть идемпотентной: повторный запуск с тем же ключом не создаёт второй пограничный кадр и не увеличивает статистики повторно.

Пограничный результат инвалидируется и рассчитывается заново, если:

изменился любой исходный H5, участвующий в пограничном кадре или 60-секундной истории;
изменился результат ProcessFileJob одного из участвующих файлов;
изменился состав background_history_signature;
между файлами был вставлен новый сосед;
файл удалён или перемещён в другую серию;
изменилась версия алгоритма или схема хранения.

При ошибке сохраняются status: failed и error_message. Внутренние результаты файлов остаются доступными, но FastAPI возвращает provisional для незавершённой границы и не использует частично записанные пограничные данные.

19. Объединение для веб-интерфейса

Физически склеивать все рассчитанные файлы в один гигантский H5 необязательно.

manifest.json является постоянным источником паспортных данных и временных отношений, которые FastAPI передаёт веб-интерфейсу. SQLite хранит индекс этих паспортов:

путь к manifest.json;
порядок файлов;
абсолютные времена;
идентификаторы непрерывных участков;
разрывы;
перекрытия;
пути к готовым артефактам.

После нажатия «Применить» сервер:

Проверяет выбор по нормативным правилам раздела 17.5.
Определяет, какие файлы пересекают выбранное время.
Читает нужные диапазоны из их готовых temporal.h5 и spectral.h5.
Расставляет данные по абсолютному времени.
При 0 ≤ Δt ≤ 1 секунды последовательно объединяет готовые массивы файлов, не создаёт отдельный объект разрыва и не изменяет временные метки исходных отсчётов.
Если выбор пересекает разрыв больше одной секунды, отклоняет запрос и возвращает описание границы из manifest.json.
Если выбор содержит наложение, использует только готовое решение OverlapResolverJob; при pending или failed совместный запрос отклоняется.

Отдельные series_temporal.h5 и series_spectral.h5 не создаются. Сервер читает готовые уровни из файловых temporal.h5 и spectral.h5, используя порядок и границы из паспортов, проиндексированных в SQLite.

20. Представление в интерфейсе

Интерфейс строится иерархически:

Серия
→ непрерывный участок
→ физические файлы

Состав отображаемых полей, допустимые комбинации файлов, уведомления и ограничения определены разделом 17.5. Интерфейс получает эти сведения через FastAPI из manifest.json и не рассчитывает временные отношения самостоятельно.

21. Фоновое разрешение временных наложений

Сравнение наложений не выполняют наблюдатель, ProcessFileJob, FastAPI или браузер. Для этого существует отдельная фоновая задача OverlapResolverJob.

21.1. Когда создаётся задача

Наблюдатель обнаруживает Δt < 0, вычисляет границы наложения и записывает в manifest.json обоих файлов:

в массиве overlaps обоих паспортов создаётся запись:
status: pending;
other_file_id;
overlap_start;
overlap_end.

Оба физических файла затем обрабатываются полностью. Исходные данные не обрезаются, а temporal.h5 и spectral.h5 создаются независимо для каждого файла.

После перехода обоих файлов в состояние ready диспетчер фоновых задач создаёт один OverlapResolverJob для этой пары и интервала. Сам наблюдатель задачу сравнения не выполняет.

Последовательность:

Watcher
→ зафиксировал overlap со статусом pending
→ оба ProcessFileJob завершены
→ OverlapResolverJob
→ duplicate или conflict
→ обновление manifest.json и SQLite
→ FastAPI читает готовое решение

Если один из исходных файлов изменился или его результат инвалидирован, готовое решение наложения также инвалидируется, status соответствующих записей overlaps возвращается в pending, а после повторной обработки задача запускается заново.

21.2. Как сравнивается наложение

Точный интервал:

overlap_start = max(start_1, start_2)
overlap_end = min(end_1, end_2)

OverlapResolverJob выполняет следующие действия:

проверяет совпадение частоты, числа каналов, порядка осей и сетки временных отсчётов;
переводит границы наложения в диапазоны отсчётов каждого файла;
читает только пересекающиеся участки Phase блоками, не превышающими установленный предел decoded-данных;
приводит оба блока к одинаковому порядку time × channel и к тому же каноническому числовому представлению, которое использует основная обработка;
сравнивает маски действительных значений;
сравнивает каждое действительное значение фазы без допуска по амплитуде;
сохраняет контрольные суммы сравниваемых канонических блоков для проверки и воспроизводимости.

Совпадение считается идеальным только тогда, когда временная сетка, маски действительности и все значения фазы для всех каналов во всём пересекающемся интервале совпадают. При несовпадении хотя бы одного значения результат считается conflict. При несовпадении сетки времени результат также считается conflict с причиной time_grid_mismatch.

Контрольная сумма ускоряет повторную проверку, но не заменяет сравнение соответствующих значений. Контрольные суммы самого H5-файла сравнивать недостаточно, потому что вне пересекающегося интервала файлы могут различаться.

21.3. Результат duplicate

duplicate означает совпадение именно пересекающегося интервала, а не обязательную идентичность двух физических файлов целиком.

В канонической временной шкале совпадающий интервал учитывается один раз. Владельцем по умолчанию становится файл, начавшийся раньше; при одинаковом времени начала используется меньший file_id.

Уникальная часть второго файла до или после наложения сохраняется и добавляется в последовательность. Физические файлы и оба комплекта temporal.h5 и spectral.h5 остаются неизменными.

Один повторяющийся граничный отсчёт обрабатывается тем же правилом: если фаза совпала, он берётся один раз и не создаёт предупреждение или видимый разрыв.

21.4. Результат conflict

conflict означает, что пересекающийся интервал содержит разные значения или несовместимую временную сетку.

Автоматически усреднять, складывать, интерполировать или смешивать результаты запрещено. По умолчанию владельцем пересекающегося интервала становится файл, начавшийся раньше; при одинаковом времени начала используется меньший file_id.

Пользователь получает предупреждение и может выбрать второй источник. Переключение передаётся как параметр конкретного запроса и меняет только читаемый готовый temporal.h5 или spectral.h5. Оно не изменяет manifest.json для других пользователей, а исходный H5 повторно не обрабатывается.

Уникальные части обоих файлов вне пересечения остаются доступными. При построении ответа пересекающийся диапазон учитывается только один раз из выбранного источника.

21.5. Что записывается после разрешения

OverlapResolverJob обновляет manifest.json обоих файлов и индекс SQLite:

в соответствующей записи массива overlaps:
status: duplicate или conflict;
other_file_id;
overlap_start;
overlap_end;
default_source_file_id;
comparison_revision;
resolved_at;
comparison_block_hashes;
conflict_reason, если имеется.

Если задача не смогла прочитать или сопоставить интервал, она записывает status: failed и error_message. Такой результат нельзя трактовать как conflict или duplicate.

Временные отношения и готовое решение должны существовать в manifest.json. SQLite только индексирует их для быстрого поиска.

21.6. Поведение FastAPI

FastAPI никогда не читает исходный Phase, не вычисляет контрольные суммы и не сравнивает значения при нажатии «Применить».

FastAPI:

при duplicate читает пересекающийся диапазон только из файла-владельца;
при conflict использует источник из параметра запроса, а при его отсутствии — default_source_file_id, и возвращает предупреждение;
при pending или failed не разрешает совместный запрос перекрывающихся файлов;
вне интервала наложения читает уникальные части каждого выбранного файла;
возвращает overlap_status, границы и фактически использованный источник вместе с данными.

WATCHER PRINCIPE:
1. Назначение наблюдателя

Наблюдатель отвечает только за:

обнаружение H5
→ проверку завершённости копирования
→ чтение паспорта
→ создание технического идентификатора
→ поиск или создание серии
→ сортировку файлов серии
→ определение временного положения
→ запись связей с соседними файлами в manifest.json
→ создание каталогов
→ постановку файла в очередь

Наблюдатель не должен:

рассчитывать графики;
сравнивать значения пересекающихся файлов;
определять, является ли наложение дубликатом;
обрезать файлы;
объединять исходные массивы.

После готовности обоих файлов диспетчер фоновых задач запускает OverlapResolverJob. Только эта задача сравнивает наложение и записывает готовое решение в manifest.json и SQLite.

2. Обнаружение нового файла

Наблюдатель реагирует на появление .h5 и дополнительно пересматривает входную папку примерно каждые 5 секунд.

Файл считается готовым, если:

его размер не изменился за две последовательные проверки;
время изменения не поменялось;
файл успешно открывается;
набор Phase найден;
форма и обязательные метаданные читаются корректно.

Состояния:

discovered
→ stabilizing
→ inspected
→ indexed
→ queued
→ processing
→ ready / failed

Папка calculated_data и временные файлы обязательно исключаются из наблюдения.

4. Названия серий и файлов

Время не входит в имя каталога. Оно хранится только в паспорте и SQLite.

Техническое имя серии:

series_0-5000m_a84f19c203bd

Техническое имя результата файла:

file_0-5000m_a23b19e4

Здесь:

0-5000m — участок кабеля;
последние символы — короткий технический идентификатор;
полная контрольная сумма хранится в паспорте.

Метраж в имени нужен для навигации, но истинным идентификатором остаётся контрольная сумма. Поэтому одинаковые участки разных серий или разные файлы одной серии не конфликтуют по именам.

Отображаемое название серии:

Кабель 0–5000 м · 1000 Гц · 3125 каналов

Пользователь может изменить отображаемое название без переименования технической директории.

5. Сопоставление файла с серией

Файл относится к существующей серии, если совпадают:

идентификатор линии или оборудования, если он существует;
начало и окончание участка кабеля;
количество каналов;
шаг между каналами;
частота;
порядок осей;
физический тип данных.

Время записи не участвует в создании серии. Поэтому одна серия может содержать записи одного кабеля за разные часы или дни.

Если идентификатора оборудования нет, для защиты от случайного объединения дополнительно учитывается исходная директория эксперимента.

6. Определение временного отношения файлов

После добавления нового файла все файлы серии сортируются по времени начала, а при одинаковом начале — по file_id.

Наблюдатель проходит список слева направо и поддерживает максимальное время окончания уже покрытого участка. Для нового файла он находит все ранее встреченные файлы, окончание которых позже его времени начала. Каждая такая пара регистрируется как отдельное наложение в массиве overlaps; проверять только соседнюю пару недостаточно.

Если активных наложений нет, Δt вычисляется относительно максимального времени окончания текущего участка по единой формуле и классификации раздела 17.

Наблюдатель выполняет только следующие действия:

при 0 ≤ Δt ≤ 1 секунды помещает файл в текущий непрерывный участок с relation: continuous и создаёт согласованные записи boundaries со status: awaiting_files в manifest.json обоих соседей;
при Δt > 1 секунды начинает новый непрерывный участок с relation: gap и записывает для этой границы status: skipped_gap;
для каждой пересекающейся пары оставляет файлы в одном непрерывном участке и записывает в overlaps status: pending, other_file_id, overlap_start и overlap_end; BoundaryFinalizeJob для такой пары не создаётся.

Наблюдатель не определяет duplicate или conflict и не читает Phase для сравнения значений.

После сортировки он обновляет manifest.json нового файла и всех затронутых предыдущих и следующих соседей. Если канонический сосед изменился, старая запись boundaries инвалидируется до запуска новой задачи. Затем индекс SQLite синхронизируется с паспортами.

Точное положительное значение Δt до одной секунды сохраняется для воспроизводимости, но не передаётся интерфейсу как отдельное предупреждение о разрыве. Исходные временные метки при этом не изменяются.

После создания паспорта наблюдатель:

Создаёт директорию результата файла.
Записывает начальный manifest.json.
Создаёт или обновляет серию и порядок файлов.

Передаёт идентификатор файла в очередь обработки.

После завершения ProcessFileJob состояние обработки и пути к готовым temporal.h5 и spectral.h5 записываются в manifest.json. Только после этого файл помечается готовым для выбора в интерфейсе. Готовность файла не означает готовность его будущих пограничных расчётов; состояния boundaries ведутся независимо.

Диспетчер фоновых задач находит все пары boundaries со status: awaiting_files, для которых оба файла готовы, переводит их в pending и ставит один BoundaryFinalizeJob на текущие версии пары.

Диспетчер фоновых задач затем находит все записи overlaps со status: pending, для которых оба файла готовы, и ставит один OverlapResolverJob на каждую пару и интервал. Повторная постановка той же версии пары запрещена.
